# Mindstrate 上下文工程优化计划

## 1. 目的

这份文档不是实现说明，而是基于你给的 DeepSeek 分享内容，对 `Mindstrate` 当前架构做一次定位，并给出一份按阶段推进的优化计划。

目标有两个：

1. 先确认我们对外部观点和当前项目现状的理解是一致的。
2. 再决定哪些建议值得做、应该先做什么、怎么验证成效。

## 2. 我对分享内容的理解

分享的核心不是“向量检索没用”，而是把 agent 记忆系统分成了两类：

- `Camp 1: Memory Backends`
  - 重点是“记住什么”
  - 典型路径是：抽取事实 -> 存储 -> 检索 -> 注入
  - 优势是简单、高性能、容易做召回评估
  - 缺点是上下文主要是隐藏式存储，对人和 agent 都不够可读、可演化

- `Camp 2: Context Substrates`
  - 重点是“让 agent 在什么上下文里工作”
  - 典型路径是：读取结构化上下文文件 -> 工作 -> 写回 -> 下次继续复用
  - 优势是上下文可读、可编辑、可积累、可版本化
  - 缺点是工程复杂度更高，若设计不好会导致上下文污染、同步复杂和维护成本上升

我认为这条分享给 `Mindstrate` 的真正启发不是“把当前 RAG 推翻重来”，而是：

- 保留当前 `Camp 1` 的强项：检索、反馈闭环、质量演化
- 补足 `Camp 2` 的关键短板：让 session / project snapshot / curated knowledge 真正变成一个可持续复用的工作上下文入口

## 3. 我对当前项目的理解

基于当前代码，我对项目的判断如下：

### 3.1 已经做得很强的部分

- `packages/server/src/retrieval/retriever.ts`
  - 已经有任务导向的 `curateContext`
  - 不只是搜知识，还会分出 `knowledge / workflows / warnings`

- `packages/server/src/mindstrate.ts`
  - 已经有 `Session Memory`
  - 已经有 `Project Snapshot`
  - 已经有 `Knowledge Evolution`
  - 已经有 `feedback loop`

- `packages/obsidian-sync`
  - 已经把一部分知识转换成 Markdown 形态
  - 这已经具备了一部分 Camp 2 的“文件原生”特征

### 3.2 当前最明显的缺口

虽然组件已经存在，但还没有形成“agent 开工前的统一上下文装配入口”。

现在的能力更像：

- `session_restore` 负责恢复会话
- `memory_curate` 负责任务相关知识包
- `project snapshot` 由 `mindstrate init` 生成，但不一定会被稳定纳入每次任务的上下文

也就是说，当前项目已经有“上下文零件”，但还没有一个足够强的一等工作流把这些零件稳定组合起来。

这正是链接里最值得优先落地的建议。

## 4. 总体判断

我的结论是：

- `Mindstrate` 当前定位最准确的说法是：
  - `以 Camp 1 为核心引擎`
  - `已经具备 Camp 2 的基础设施`
  - `但还没有形成 Camp 2 风格的主工作流`

- 所以优化方向不应该是：
  - 立刻把 SQLite/向量索引废掉
  - 立刻把 Markdown 变成唯一事实源
  - 立刻上复杂知识图谱或 dream cycle

- 更合理的方向应该是：
  - 先补“上下文装配入口”
  - 再补“时序与衰减”
  - 然后再考虑“文件真相源”和“后台整理机制”

## 5. 优化原则

这次优化建议遵循下面四条原则：

1. 不推翻当前架构
   - 现有 `search / feedback / evolve / session / snapshot` 都是资产，不是包袱。

2. 优先补工作流，不优先补存储革命
   - 先让 agent 更稳定地“读上下文 -> 工作 -> 写回”。

3. 每一步都要可验证
   - 不能只加概念层功能，要能用测试和使用路径证明它变好。

4. 先做低风险高收益项
   - 优先做接口层、编排层、评分层的小步升级。

## 6. 分阶段计划

## 阶段 0：设计校准

目标：

- 把“我们到底要把 Mindstrate 推向哪一种 Camp 2 能力”说清楚

要产出：

- 一份确认后的优化方向
- 一份明确的非目标列表

建议结论：

- 短期目标不是“文件真相源”
- 短期目标是“统一上下文装配”
- 中期目标是“时序感知 + 自动整理”
- 长期目标才是“文件原生真相源 + 可移植上下文包”

验收标准：

- 你认可本文档中的总体判断和阶段排序

## 阶段 1：把上下文装配做成一等入口

这是我建议最先做的阶段。

### 6.1 目标

让 agent 在开始复杂任务时，不再靠“分别调用多个工具、自己拼接上下文”，而是直接得到一个统一工作上下文。

### 6.2 要做什么

- 新增一个高层能力，暂定名：
  - `context_assemble`
  - 或保守一点，增强现有 `memory_curate`

- 该能力至少要组合以下信息：
  - 最近会话摘要
  - 未完成任务
  - 项目快照
  - 当前任务的 curated knowledge/workflows/warnings

- 在 MCP 层和 Team Server 层同时暴露

- 输出格式要可直接给 agent 使用，而不是只返回原始对象

### 6.3 为什么先做这个

- 这是分享里最贴合当前项目、改动最小、收益最高的一步
- 它不要求重做存储层
- 它能直接把现有 `Session Memory + Project Snapshot + Curate` 从“散件”变成“工作流”

### 6.4 涉及模块

- `packages/server/src/mindstrate.ts`
- `packages/server/src/retrieval/retriever.ts`
- `packages/protocol/src/models/knowledge.ts`
- `packages/team-server/src/server.ts`
- `packages/client/src/team-client.ts`
- `packages/mcp-server/src/tools/*`

### 6.5 验收标准

- agent 可以通过一次调用拿到完整工作上下文
- 本地模式和团队模式返回的数据结构一致
- 有单元测试覆盖：
  - 有 session 但无 snapshot
  - 有 snapshot 但无 session
  - session + snapshot + curated 全部存在
  - 完全空上下文时的兜底表现

## 阶段 2：引入时序意识，而不是只有静态质量分

### 6.6 目标

让知识不只按“相关性 + 质量分 + 反馈”排序，还能体现“是否过时”。

### 6.7 要做什么

- 为知识检索增加时间维度
  - 区分“最近仍有效”与“历史上曾有效但可能过期”

- 增加更明确的过期策略
  - 框架版本相关知识更快衰减
  - 项目约定类知识更慢衰减

- 在评分时使用更细分的时序规则，而不是统一线性衰减

### 6.8 为什么这一步排第二

- 分享里提到的 `Supermemory` 式时序感知，对编程知识非常重要
- 但它是排序策略升级，不要求先改存储模型
- 风险低于“改真相源”

### 6.9 验收标准

- 检索结果能区分近期知识与疑似陈旧知识
- 有测试覆盖不同类型知识的衰减策略
- 至少增加一项评估指标，观察时序规则是否改善结果质量

## 阶段 3：做轻量级后台整理，而不是一上来就 dream cycle

### 6.10 目标

让知识库不只在写入时处理，也能在后台周期性整理。

### 6.11 要做什么

- 先做轻量版，不直接做复杂 dreaming
- 第一版后台整理任务建议只做三件事：
  - 检测重复或高度相似知识
  - 标记长期低采纳/高拒绝知识
  - 汇总高频使用知识，生成待改进建议

- 优先复用现有：
  - `memory_evolve`
  - `feedback loop`
  - `quality scorer`

### 6.12 为什么不直接上 full dream cycle

- 当前项目还没有上下文污染控制、版本冲突合并、图谱层支撑
- 直接做复杂 dream cycle，工程风险高，容易先把系统搞脆

### 6.13 验收标准

- 后台任务可以定期运行，不阻塞主流程
- 不自动改写高风险知识
- 输出可审计的建议列表

## 阶段 4：把 Markdown 从“镜像”推进到“协作副本”

### 6.14 目标

在不推翻现有 SQLite 主存的前提下，先让 Markdown 更接近协作主界面。

### 6.15 要做什么

- 明确哪些字段可以在 Markdown 中安全编辑
- 补齐冲突检测与同步语义
- 让项目快照、工作流、约定类知识在 Obsidian 中更容易人工维护

### 6.16 为什么先不做“Markdown 唯一真相源”

- 这一步牵涉：
  - 版本管理
  - 冲突合并
  - 索引增量更新
  - 团队同步语义
  - 权威来源定义

- 如果阶段 1-3 还没跑稳，直接做这一步容易把系统复杂度拉爆

### 6.17 验收标准

- 关键知识类型在 Markdown 中可安全编辑
- 不出现明显的 SQLite / Markdown 状态漂移
- 用户能理解“哪些编辑会回写，哪些只是私人笔记”

## 阶段 5：评估是否需要真正的文件原生真相源

### 6.18 目标

只有在前面几阶段跑通后，再判断是否值得把 Markdown 提升为 canonical source of truth。

### 6.19 进入这一阶段前必须先回答的问题

- 团队协作时到底谁是权威来源？
- Team Server 与本地 Vault 的冲突怎么解？
- 版本历史由 Git 管，还是由数据库管，还是双轨？
- 现有检索、反馈、演化流程如何从“主存数据库”迁移为“文件主存 + 索引影子层”？

### 6.20 我的建议

在没有真实使用压力之前，不建议立刻做这一步。

## 7. 不建议现在就做的事

下面这些方向不是永远不做，而是不建议现在就做：

- 完整知识图谱化
  - 当前收益不一定覆盖实现复杂度

- 全量声明式依赖图引擎
  - 很酷，但现在偏超前设计

- 一次性重写检索管线为多步 DAG 深搜
  - 可以作为实验方向，但不该先于“统一上下文入口”

- Markdown 成为唯一真相源
  - 现在缺少冲突语义和版本治理方案

## 8. 建议的实施顺序

建议按下面顺序推进：

1. `阶段 1：统一上下文装配入口`
2. `阶段 2：时序意识`
3. `阶段 3：轻量后台整理`
4. `阶段 4：Markdown 协作副本增强`
5. `阶段 5：评估文件原生真相源`

## 9. 如果让我开始实施，我会怎么拆第一阶段

第一阶段我会再细分成下面几个可交付小步：

1. 定义新的上下文装配返回结构
   - 不急着决定工具名，先明确数据契约

2. 在 server facade 层实现装配逻辑
   - 组合 `session + snapshot + curate`

3. 打通 team-server / client / mcp-server
   - 保证本地模式和团队模式行为一致

4. 增加测试
   - 先验证上下文拼装的正确性，再考虑输出格式

5. 再决定是否对现有 `memory_curate` 做兼容增强
   - 避免一开始就破坏已有调用方

## 10. 本文档对应的结论

一句话总结：

`Mindstrate` 最值得做的优化，不是立刻从 Camp 1 彻底改造成 Camp 2，而是先把已有的 session、snapshot、curation 组合成稳定的一等上下文入口；等这条主工作流跑顺，再逐步补时序感知、后台整理和文件原生协作能力。
